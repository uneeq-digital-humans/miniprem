# EKS Terraform Configuration

This directory contains the Terraform infrastructure-as-code for deploying MiniPrem on Amazon EKS (Elastic Kubernetes Service).

## Directory Structure

```
terraform/eks/
├── main.tf                  # Main Terraform configuration and provider setup
├── eks.tf                   # EKS cluster configuration
├── vpc.tf                   # VPC and networking resources
├── node-groups.tf           # EKS node groups (GPU and control plane)
├── iam.tf                   # IAM roles and policies
├── variables.tf             # Input variables
├── outputs.tf               # Output values
├── terraform.tfvars         # Configuration values (environment-specific)
├── terraform.tfvars.example # Example configuration template
├── ubuntu_eks_userdata.sh   # User data script for Ubuntu EKS nodes
└── ubuntu_userdata.sh       # Additional userdata configuration
```

## Prerequisites

1. **AWS CLI** configured with appropriate credentials
2. **Terraform** v1.0+ installed
3. **kubectl** for Kubernetes cluster management
4. **helm** for application deployment

## Configuration

### terraform.tfvars

This is the single source of truth for your deployment configuration. Key variables:

```hcl
# AWS Configuration
aws_region = "us-east-2"

# Cluster Configuration
project_name = "renny"
environment = "production"
deployment_id = ""  # Auto-generated by deploy.sh

# Node Configuration
renny_instance_type = "g5.4xlarge"
renny_min_size = 10
renny_max_size = 20
renny_desired_size = 10

# UneeQ/DHOP Credentials
dhop_tenant_id = "your-tenant-id"
dhop_api_key = "base64-encoded-api-key"

# Harbor Registry (contact help@uneeq.com for robot account)
harbor_username = "robot$your-customer-name"
harbor_password = "your-harbor-robot-password"
```

## Usage

**DO NOT run Terraform commands directly from this directory!**

Instead, use the wrapper scripts in `kubernetes/scripts/`:

```bash
# Deploy infrastructure
cd /path/to/miniprem-2025/kubernetes
./scripts/deploy.sh

# Check deployment status
./scripts/status.sh

# Scale Renny instances
./scripts/scale.sh 15

# Destroy infrastructure
./scripts/destroy.sh
```

The wrapper scripts handle:
- Deployment ID management
- Path resolution
- AWS profile configuration
- Error handling
- Post-deployment tasks

## Direct Terraform Commands (Advanced)

If you must run Terraform directly:

```bash
cd kubernetes/terraform/eks/

# Initialize Terraform
terraform init

# Validate configuration
terraform validate

# Plan changes
terraform plan

# Apply changes
terraform apply

# Destroy infrastructure
terraform destroy
```

## Key Resources Created

- **EKS Cluster**: Kubernetes control plane
- **VPC**: Isolated network with public/private subnets across 3 AZs
- **NAT Gateways**: One per AZ for private subnet internet access
- **Node Groups**:
  - Renny GPU nodes (g5.4xlarge with Ubuntu)
  - Control plane nodes (t3.large with Ubuntu)
- **IAM Roles**: For cluster, nodes, and service accounts
- **Security Groups**: Network access controls

## GPU Configuration

GPU nodes use:
- **AMI**: Ubuntu EKS 22.04 (Jammy)
- **Instance Types**: g5.4xlarge (default) with NVIDIA A10G GPUs
- **GPU Time-Slicing**: Configured via `kubernetes/values/renny-values.yaml`
- **Driver Installation**: Automatic via GPU Operator

## Troubleshooting

### Terraform State Issues

If you encounter state lock issues:

```bash
# List Terraform state
terraform state list

# Show specific resource
terraform state show <resource>

# Remove stuck resource
terraform state rm <resource>
```

### Path Issues

After reorganization, all paths now use:
- `TERRAFORM_DIR="$PROJECT_DIR/kubernetes/terraform/eks"`
- Scripts automatically resolve relative paths

### Provider Issues

If providers are missing after the reorganization:

```bash
cd kubernetes/terraform/eks/
rm -rf .terraform .terraform.lock.hcl
terraform init
```
